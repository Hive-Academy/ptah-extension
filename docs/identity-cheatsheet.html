TabId & SessionId — End-to-end

How identifiers flow from the renderer through the JSON-RPC wire into the Claude SDK and back. Includes every file you need to read, in the order you should read them.

1. Brand types

All identifiers are branded strings defined in two places:

  • libs/shared/src/lib/types/branded.types.ts — cross-platform brands shared by frontend + backend
  • libs/frontend/chat-state/src/lib/identity/ids.ts — frontend-only brands + re-export of shared TabId

TabId

UI surface (tile in navbar/canvas). Lives only in the renderer. Never round-trips through the SDK. Used as the routing key between renderer ↔ backend until the real SDK UUID is known.

Format: UUID v4. Minted by TabId.create().

SessionId

The SDK's UUID, emitted by the Claude SDK in the system: 'init' message. Canonical session identifier everywhere. Never mint a Ptah-side SessionId — only the SDK does.

Format: UUID v4. Brand re-exported on the frontend as ClaudeSessionId.

ConversationId

User-perceived thread that survives compaction and multi-tab fan-out (canvas grid mode). One conversation contains an ordered list of ClaudeSessionIds.

Frontend-only. Minted by ConversationId.create().

SurfaceId

Sibling to TabId for non-tab streaming consumers — wizard analysis phases, harness builder operations. Disjoint key space so tab enumeration never accidentally picks up wizard/harness state.

BackgroundAgentId

Sub-agent spawned inside a session. Distinct brand so the background-agent store can never collide with parent-session keys.

MessageId · CorrelationId

Per-message identifiers used for streaming chunk reassembly and RPC request/response correlation. Same UUID v4 invariant.

Smart-constructor contract

// libs/shared/src/lib/types/branded.types.ts
export const TabId = {
  create():    TabId      // fresh uuidv4() as TabId
  validate(s): s is TabId // /^[0-9a-f]{8}-...$/i.test(s)
  from(s):     TabId      // throws TypeError on invalid UUID
  safeParse(s):TabId|null // null on invalid UUID
}
// Same shape for SessionId, MessageId, CorrelationId, JobId, RunId.
⚠ Invariant .from() THROWS on non-UUID input. Any code path that mints non-UUID values and feeds them into a .from() validator will crash. This is the v0.2.32 regression class — see Known minting bombs.

5. SDK system init message — the only time SessionId is created

The Claude Agent SDK is the sole authority for SessionId. Ptah never mints one. The system 'init' message looks like:

{
  type: 'system',
  subtype: 'init',
  session_id: "e1b2c3d4-...",   // ← canonical SessionId
  cwd: "D:/projects/foo",
  model: "claude-sonnet-4-6",
  permissionMode: "acceptEdits",
  tools: [...], mcp_servers: [...], slash_commands: [...]
}

This is consumed by SdkMessageTransformer at libs/backend/agent-sdk/src/lib/sdk-message-transformer.ts, which:

  1. Casts msg.session_id as SessionId (wire-boundary cast — safe by SDK contract).
  2. Notifies SessionLifecycleManager to call SessionRegistry.bindRealSessionId(tabId, sessionId).
  3. Emits a FlatStreamEventUnion carrying both tabId and sessionId.

7. Stream routing — how an event finds its tab

Mental model Backend speaks sessionId on the wire. Frontend speaks tabId in the UI. The router is the only place those two meet. 
// libs/frontend/chat-routing/src/lib/stream-router.service.ts

routeStreamEvent(event):
  sessionId = event.sessionId as ClaudeSessionId        // wire boundary
  convId    = ConversationRegistry.findBySession(sessionId)
  tabs      = TabSessionBinding.tabsFor(convId)         // 1..N tabs
  for tab of tabs:
    StreamingHandler.append(tab, event)                 // fan-out

Permission prompts and AskUserQuestion requests carry both tabId and sessionId on the wire (Zod-refined to UUID on receive). The router prefers tabId first, falls back to sessionId → convId → tabs.

9. Compaction — why ConversationId exists

When the SDK auto-compacts, it may emit a compact_boundary message carrying a new session_id. Without ConversationId the tab would lose its thread. The flow:

  1. SdkMessageTransformer sees compact_boundary, emits a compaction_start FlatEvent.
  2. ConversationRegistry.setCompactionState(convId, { inFlight: true }).
  3. Next system init brings the new SessionId; backend calls bindRealSessionId(tabId, newSessionId) — the SAME tab record now has TWO bySessionId entries.
  4. Frontend appends the new sessionId to ConversationRecord.sessions[]; TabState.claudeSessionId updates to the head session.
  5. compaction_complete clears the inFlight flag.

Files: libs/frontend/chat/src/lib/services/chat-store/compaction-lifecycle.service.ts

11. SessionRegistry — the backend dual-index

libs/backend/agent-sdk/src/lib/helpers/session-lifecycle/session-registry.service.ts

// Both maps point at the SAME SessionRecord object.
// Mutations via either lookup are immediately visible from the other.
class SessionRegistry {
  private byTabId     = new Map<string, SessionRecord>()
  private bySessionId = new Map<string, SessionRecord>()

  register(tabId, config, abortCtl): SessionRecord       // inserts into byTabId only
  bindRealSessionId(tabId, realSessionId): void          // adds bySessionId entry, no rekey
  find(idOrTabId): SessionRecord | undefined           // checks both, O(1)
  remove(rec): void                                      // deletes from both indexes
  setSessionQuery(sessionId, query): void
  getActiveSessionIds(): SessionId[]                     // most-recent-first
  markActive(tabId): void
  clearAll(): void
}
Design property No rekey, no scanning, no shifting-key semantics. Callers stop caring whether they hold a tabId or a sessionId — find() resolves both.
Introduced by TASK_2026_118, replacing the legacy tabIdToRealId side-map + findByTabOrRealId reverse-scan helpers.

SessionRecord fields

interface SessionRecord {
  readonly tabId:       string           // immutable, assigned at register()
           realSessionId: string | null    // null until bindRealSessionId — set ONCE
           query:         Query  | null    // SDK Query handle
  readonly config:         AISessionConfig
  readonly abortController:AbortController
           messageQueue:  SDKUserMessage[]
           resolveNext:   (() => void) | null
           currentModel:  string
}

12. TabSessionBinding + ConversationRegistry — the frontend dual-index

libs/frontend/chat-state/src/lib/tab-session-binding.service.ts · .../conversation-registry.service.ts

TabSessionBinding:
  TabId     ──▶ ConversationId   // _byTab
  ConversationId ──▶ Set<TabId>  // _byConversation (reverse)
  SurfaceId ──▶ ConversationId   // _bySurface (disjoint key space)

ConversationRegistry:
  ConversationId ──▶ {
    sessions: ClaudeSessionId[],   // oldest-first, head = current SDK session
    compactionInFlight: boolean,
    lastCompactionAt: number | null,
    ...
  }

The two services are deliberately split: TabSessionBinding owns the TabId ↔ ConversationId edge; ConversationRegistry owns the ConversationId → SessionId[] list. StreamRouter is the only consumer that composes them.

14. Known minting bombs (audit gaps)

Regression class Any production code that mints a non-UUID string and feeds it into SessionId.from(), TabId.from(), or a Zod .refine(UUID_REGEX) schema will throw. The audit in docs/identity-audit.md classifies these but does not remediate them.

3 libs/backend/rpc-handlers/src/lib/harness/streaming/harness-stream-broadcaster.service.ts:91 harness-${operationId} Synthetic; needs own brand (HarnessStreamId)

5 libs/backend/agent-generation/src/lib/services/content-generation.service.ts:447 gen-${agentId} Internal scope, lower risk
6 libs/frontend/chat/src/lib/services/chat-store/conversation.service.ts:318 '' as SessionId Sentinel in ExecutionChatMessage
7 libs/frontend/chat/src/lib/services/message-sender.service.ts:407 '' as SessionId Same sentinel pattern

Recommended remediation order

  1. Narrow TabState.id to TabId and TabState.claudeSessionId to SessionId | null. Eliminates ~15 questionable casts at once.
  2. Replace '' as SessionId sentinels with SessionId | null + explicit null handling.
  3. Introduce HarnessStreamId and WizardPhaseId brands for the synthetic IDs; stop routing them through SessionId-typed APIs.
  4. Add an ESLint rule forbidding as <BrandType> outside branded.types.ts / ids.ts / test files.
