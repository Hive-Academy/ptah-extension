# SessionId Research Report

## Executive Summary

This report documents the comprehensive analysis of `sessionId` usage across the Ptah Extension codebase. The investigation reveals a **two-phase sessionId lifecycle**: a placeholder "draft" ID generated by the frontend and a real SDK UUID resolved by the backend. The report identifies all generation points, flow paths, and usage patterns to inform architectural decisions.

## Key Findings

### 1. SessionId Is Generated in TWO Places

| Location                           | Format                                   | Purpose                                       | Lifecycle                                                    |
| ---------------------------------- | ---------------------------------------- | --------------------------------------------- | ------------------------------------------------------------ |
| **Frontend (ConversationService)** | `msg_{timestamp}_{random7chars}`         | Placeholder for new conversations             | Created before RPC call, replaced when SDK responds          |
| **Backend (Claude Agent SDK)**     | Real UUID from SDK system 'init' message | Actual session identifier for SDK persistence | Permanent - stored in `~/.claude/projects/{sessionId}.jsonl` |

## Detailed Analysis

### GENERATION POINT 1: Frontend Placeholder Generation

**File**: `libs/frontend/chat/src/lib/services/chat-store/conversation.service.ts`

```typescript
// Line 70-72
private generateId(): string {
  return `msg_${Date.now()}_${Math.random().toString(36).substring(2, 9)}`;
}

// Line 281
const sessionId = this.generateId();
```

**Purpose**: Creates a temporary identifier to:

1. Track the RPC request before SDK responds
2. Allow immediate UI updates (show user message)
3. Provide a correlation key for SessionManager state machine

**Lifecycle State**:

- Set with state `'draft'` via `SessionManager.setSessionId(sessionId, 'draft')`
- Transitions to `'confirmed'` when real SDK UUID arrives

---

### GENERATION POINT 2: Real SDK UUID Resolution

**File**: `libs/backend/agent-sdk/src/lib/helpers/stream-transformer.ts`

```typescript
// Lines 206-216
if (isSystemInit(sdkMessage)) {
  const realSessionId = sdkMessage.session_id;

  // Notify caller of the real session ID
  if (onSessionIdResolved) {
    onSessionIdResolved(realSessionId);
  }
}
```

**Source**: The Claude Agent SDK sends a system 'init' message containing the real UUID:

```typescript
interface SDKSystemInit {
  type: 'system';
  subtype: 'init';
  session_id: string; // Real SDK UUID
  // ...
}
```

**SDK Storage**: Real session ID maps to file `~/.claude/projects/{sessionId}.jsonl`

### WHERE SESSION ID IS REQUIRED BY SDK

The Claude Agent SDK **requires** a real session UUID for:

1. **Session Resume** (`options.resume`):

   ```typescript
   // sdk-agent-adapter.ts line 756
   resumeSessionId: sessionId as string;
   ```

2. **Message Persistence** (SDK native):

   - SDK stores messages to `~/.claude/projects/{sessionId}.jsonl`
   - Uses session_id from system 'init' message

3. **Session Query Object**:
   ```typescript
   // SessionLifecycleManager line 103
   setSessionQuery(sessionId: SessionId, query: Query): void
   ```

---

### WHERE SESSION ID IS USED FOR FRONTEND CORRELATION

**These usages are for UI routing, NOT SDK requirements:**

| Location                                 | Usage                       | Could Use TabId Instead?              |
| ---------------------------------------- | --------------------------- | ------------------------------------- |
| `TabManagerService.findTabBySessionId()` | Find tab by session         | Yes - if tabs had stable IDs          |
| `TabState.claudeSessionId`               | Link tab to session         | Yes - could use mapping               |
| `StreamingState` event routing           | Route events to correct tab | Yes - events could carry tabId        |
| `processStreamEvent()` initialization    | Initialize tab sessionId    | Necessary - this IS the linking point |

## Critical Observations

### 1. Placeholder vs Real ID Timing Gap

There is a timing gap between:

- Frontend generating `msg_XXX` placeholder
- Backend SDK returning real UUID in system 'init' message

**Current Solution**: `StreamingHandlerService.processStreamEvent()` handles this by checking:

```typescript
if (!targetTab.claudeSessionId && (activeTab.status === 'fresh' || activeTab.status === 'streaming' || activeTab.status === 'draft')) {
  // Initialize session ID from first event
  this.tabManager.updateTab(activeTab.id, { claudeSessionId: event.sessionId });
}
```

### 2. Tab ↔ Session Relationship

Currently: **Tab finds session by claudeSessionId**

```typescript
findTabBySessionId(sessionId: string): TabState | null {
  return this.tabs().find((t) => t.claudeSessionId === sessionId) ?? null;
}
```

Alternative: **Events carry tabId, skip lookup**

- Would require backend to track tabId and include in events
- More complex but eliminates sessionId lookups on frontend

### 3. SessionManager State Machine

The SessionManager tracks session state:

```typescript
type SessionState = 'draft' | 'confirming' | 'confirmed' | 'failed';
```

- **draft**: Placeholder created, awaiting SDK
- **confirmed**: Real SDK UUID received
- **failed**: Session creation failed

## Recommendations

### Option A: Keep Current Architecture (Recommended)

**Pros**:

- Works correctly today
- SDK sessionId is the universal identifier
- Simple mental model: sessionId = session

**Cons**:

- Minor timing complexity on first event

**Improvements**:

1. Document the placeholder → real ID flow explicitly
2. Add diagnostic logging for session ID transitions
3. Consider storing both `placeholderId` and `realSessionId` for debugging

### Option B: Add TabId to Events

**Pros**:

- Eliminates sessionId lookup on frontend
- Clearer event → tab routing

**Cons**:

- Requires backend to track tabId
- Adds another identifier to manage
- SDK events don't naturally have tabId

**Implementation**:

1. Frontend sends `tabId` with RPC calls
2. Backend associates tabId with session
3. Backend includes tabId in outbound events
4. Frontend routes events by tabId instead of sessionId

### Option C: Generate Real IDs on Frontend

**Not Recommended**:

- SDK generates its own session IDs
- Would create ID mismatch with SDK storage
- Fundamentally incompatible with SDK architecture
