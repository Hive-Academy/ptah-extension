# `.ptah/roadmap.md` Schema

The roadmap is the single source of truth that connects Stage A (foundation) to Stage B (per-module implementation). Each unchecked item becomes its own chat session. Stage B sessions read this file to pick up context and dependencies, so the schema must be parseable and stable.

## File location

Always `.ptah/roadmap.md` at the workspace root. Never rename or move.

## Top-level structure

```markdown
# Roadmap — <Project Name>

> Generated by saas-workspace-initializer (Stage A).
> Each unchecked item is a Stage B task. Run `/orchestrate <slug>` in a new chat session.

## Phase 1: Foundation

- [x] <slug>: <title>
      ...

## Phase 2: <name>

- [ ] <slug>: <title>
      ...
```

## Item schema

Every item is a checklist entry followed by an indented charter and dependency line.

```markdown
- [ ] <slug>: <title>
      **Charter.** <one paragraph: what it builds, why it matters, what "done" looks like — 3-5 sentences>
      **Depends on:** <slug>, <slug> (or `none` for entry-point items)
```

### Field rules

- **Checkbox**: `[x]` for items completed by Stage A foundation; `[ ]` for everything else.
- **Slug**: kebab-case, unique across the file, stable. Stage B sessions reference items by slug. Examples: `tenant-context`, `user-auth-jwt`, `billing-stripe`, `webhook-stripe-subscriptions`.
- **Title**: human-readable, under 60 chars.
- **Charter**: one paragraph. Must answer (1) what library/module/feature gets built, (2) why this matters for the SaaS, (3) the success signal (a passing test, a working endpoint, a deployed pipeline). No implementation steps — those belong in the Stage B task.
- **Depends on**: comma-separated list of prior slugs. Use `none` for items that need only the foundation. Cycles are forbidden.

## Phase grouping

Phases are level-2 headings (`## Phase N: <name>`). Standard phases, in order:

1. **Foundation** (always present, all `[x]` after Stage A)
2. **Tenancy & Auth** (if multi-tenant or external auth)
3. **Domain Modules** (one item per bounded context from discovery)
4. **Billing & Monetization** (if billing in MVP)
5. **Integrations & Webhooks** (if external integrations in MVP)
6. **Resilience & Events** (if real-time/SSE/async side-effects in MVP)
7. **Deployment & Hardening** (always present)
8. **QA & Launch** (always present)

Drop any phase that discovery proved out of scope. Do not invent extra phases unless the project clearly warrants them.

## Full example

```markdown
# Roadmap — Acme Project Tracker

> Generated by saas-workspace-initializer (Stage A).
> Each unchecked item is a Stage B task. Run `/orchestrate <slug>` in a new chat session.

## Phase 1: Foundation

- [x] workspace-init: Nx workspace + Angular + NestJS apps
      **Charter.** Bootstrap the Nx monorepo with the Angular preset, an `apps/web` Angular app, and an `apps/api` NestJS app. Configure ESLint module boundaries, Prettier, and Jest. Success: `nx run-many -t lint,typecheck,test` passes on a clean clone.
      **Depends on:** none

- [x] shared-primitives: shared/domain + api-interfaces libraries
      **Charter.** Create `libs/shared/domain` (base entity, value-object base, result type) and `libs/api-interfaces` (DTO contracts). These are imported by every Stage B domain module. Success: both libraries build and are referenced by `apps/api`.
      **Depends on:** workspace-init

- [x] tenant-baseline: Prisma + ZenStack with Tenant + User models
      **Charter.** Multi-tenant from day one requires a tenant context primitive in the foundation. Add a Prisma library, an initial `schema.prisma` with `Tenant` and `User`, ZenStack policies stubbed for row-level isolation, and a tenant-context middleware that reads `x-tenant-id`. Success: a request with no tenant header returns 400; with a valid header, the Prisma client filters by tenant.
      **Depends on:** workspace-init, shared-primitives

## Phase 2: Tenancy & Auth

- [ ] auth-clerk: Clerk authentication integration
      **Charter.** Wire Clerk as the auth provider behind the `AuthProvider` interface in `libs/shared/infrastructure/auth`. Add JWT verification middleware, a `@CurrentUser()` decorator, and a sign-in flow on the Angular app. Success: a protected `/api/me` endpoint returns the authenticated user and 401s without a Clerk token.
      **Depends on:** tenant-baseline

- [ ] tenant-provisioning: Tenant onboarding flow
      **Charter.** When a new user signs up via Clerk, create a `Tenant` row, assign the user as owner, and seed default settings. Expose `POST /api/tenants` for explicit provisioning and a Clerk webhook handler for `user.created`. Success: signing up a new user creates exactly one tenant and one membership.
      **Depends on:** auth-clerk

## Phase 3: Domain Modules

- [ ] domain-projects: Projects bounded context
      **Charter.** Create `libs/projects/{domain,application,infrastructure,feature}` following the DDD layout. Implement CRUD with tenant-scoped queries. Success: authenticated users can create, list, update, and delete projects scoped to their tenant.
      **Depends on:** tenant-provisioning

- [ ] domain-tasks: Tasks bounded context
      **Charter.** Create `libs/tasks/{domain,application,infrastructure,feature}` with task assignment, status transitions, and project association. Success: tasks belong to a project, transitions are validated, and tenant isolation holds.
      **Depends on:** domain-projects

## Phase 4: Deployment & Hardening

- [ ] deploy-docker: Multi-stage Docker + docker-compose
      **Charter.** Add `Dockerfile` (multi-stage), `Dockerfile.dev`, `docker-compose.yml` with PostgreSQL, webpack externals, and a `/api/health` endpoint. Apply production hardening (rate limiting, CORS, validation pipe). Success: `docker compose up` boots the app and `/api/health` returns 200.
      **Depends on:** domain-tasks

- [ ] deploy-ci: CI/CD pipeline
      **Charter.** Extend the foundation CI with a deploy job that runs `prisma migrate deploy` and pushes the Docker image to the chosen registry. Success: a merge to `main` produces a deployable image tagged with the commit SHA.
      **Depends on:** deploy-docker

## Phase 5: QA & Launch

- [ ] qa-smoke: End-to-end smoke suite
      **Charter.** Add Playwright smoke tests covering signup, tenant creation, and core domain CRUD. Wire into CI as a required check. Success: smoke suite runs green against a freshly seeded environment.
      **Depends on:** deploy-ci
```

## Linting rules for the writer

- No empty charters. If you cannot write 3 sentences about why an item matters, it is not a roadmap item — fold it into another item.
- No duplicate slugs.
- Every `Depends on:` slug must exist earlier in the file.
- Foundation items are `[x]` and reflect what Stage A actually scaffolded — never aspirational.
- Do not list time estimates, story points, or assignees. The roadmap is scope, not schedule.
